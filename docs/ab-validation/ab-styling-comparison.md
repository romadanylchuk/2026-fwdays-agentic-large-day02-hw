## SCSS, colocation, глобальні стилі, CSS-in-JS

| Критерій | A | B |
|----------|---|---|
| Colocated `*.scss` поруч із `.tsx` | Так (`import "./DevBuildHint.scss"`) | Так, той самий патерн |
| CSS-in-JS для фічі | Ні | Ні |
| Глобальні entry-стилі (`excalidraw-app/index.scss` тощо) для підказки | Не змінювалися | Не змінювалися |
| Вкладеність у SCSS | Під `.excalidraw-app { .dev-build-hint { … } }` | Те саме |
| Токени / змінні | `var(--color-on-surface, …)`, `var(--color-surface-low, …)` з fallback | Те саме |

**Висновок по стилях:** підхід **ідентичний** — SCSS поруч із компонентом, без другого стеку стилів і без переносу фічі в глобальні entry-файли.

## Висновок: чи правило вплинуло на результат і як

У цьому експерименті **активне `styling.mdc` не проявилося як відмінність результату** порівняно з сесією без правила: обидва варіанти дали ту саму комбінацію файлів, colocated SCSS і відсутність CSS-in-JS.

**Чому так можливо:**

1. **Промпт P сам явно вимагає** colocated `*.scss` і заборону CSS-in-JS для фічі — основні обмеження дублюють зміст правила.
2. Поведінка моделі в обох сесіях **узгоджена з критеріями `styling.mdc`** (див. таблиці відповідності в A і B); у B це зафіксовано як збіг з A «за структурою та стилями», навіть коли правило не було активним у Cursor.

**Практичний висновок:** для цієї задачі правило **не змінило спостережуваний результат**; воно могло б мати сильніший ефект у сценаріях, де промпт **не** нав’язує стилістичні обмеження або де модель схильна до іншого стеку — тут такого розбіжного сценарію не виникло.
